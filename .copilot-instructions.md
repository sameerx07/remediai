# Copilot Instructions for RemediAI

You are helping build RemediAI, an enterprise AI remediation platform.

---

## Project Documentation

Before writing code, read these files to understand current state:

- `README.md` — project overview, architecture, workflow
- `SPEC.md` — functional requirements and acceptance criteria
- `ARCHITECTURE.md` — system design, services, data flow, DB schema
- `AGENT_DESIGN.md` — agent pipeline, state contract, prompt locations, audit trail
- `TECH_STACK.md` — approved libraries and versions
- `SECURITY_GUARDRAILS.md` — security rules, identity model, PII handling
- `ROADMAP.md` — milestone order and what is in scope now

---

## Project Goal

Build a scalable Python-based agentic platform that reads .NET application exceptions from Azure Monitor / Application Insights, analyzes root cause, creates Azure DevOps Bugs, and later generates pull requests with human approval.

---

## Architecture Principles

- Use Python FastAPI for backend APIs.
- Use Python workers for background processing.
- Use LangGraph for agent workflow orchestration.
- Use Azure AI Foundry / Azure OpenAI for LLM calls.
- Use Azure Service Bus for asynchronous incident processing.
- Use PostgreSQL for incidents, reports, audit, and remediation status.
- Use Azure AI Search for RAG over source code, docs, runbooks, and past fixes.
- Use Azure DevOps REST APIs for repos, work items, branches, commits, and PRs.
- Use Key Vault and Managed Identity for secrets and authentication.
- Use clean architecture and modular package boundaries.

---

## MVP Build Order

Do not implement PR creation first.

Build in this order:

1. Project structure and domain models (Pydantic)
2. PostgreSQL schema and Alembic migrations
3. Azure Monitor KQL connector
4. Incident ingestion service and Service Bus publisher
5. Triage agent
6. Root cause agent
7. Code context agent
8. RAG retrieval agent
9. Fix planner agent
10. Azure DevOps Bug integration
11. FastAPI dashboard endpoints
12. React dashboard

---

## Coding Standards

- Use type hints on all function signatures.
- Use Pydantic v2 models for all data structures.
- Use SQLAlchemy 2.0 async ORM; no raw SQL except migrations.
- Use dependency injection where practical.
- Keep integrations isolated in `packages/integrations/`.
- Keep agent prompts versioned in `docs/prompts/`.
- Write unit tests for business logic.
- Write integration tests for Azure connectors using mock clients.
- Never hardcode secrets — use `pydantic-settings` and environment variables.
- Use `structlog` for structured logging with `correlation_id` and `incident_id` in every record.
- Run `ruff` (lint + format) and `mypy --strict` before finishing.

---

## Safety Rules

- Never auto-merge PRs.
- Never directly modify production.
- Always require human approval for code changes.
- Mask PII (emails, IPs, user identifiers) before sending any payload to an LLM.
- Store all agent decisions in the `audit_log` table with input/output summaries.
- Never expose tools to the LLM that execute shell commands or make unauthenticated HTTP calls.

## Next Best Action — Always Suggest What to Do Next

After completing any task, always end your response with a **"What's Next"** block in this exact format:

```
## What's Next

**Option 1 — [Short title] ✅ Recommended**
[One sentence explaining why this is the best next step given current progress.]

**Option 2 — [Short title]**
[One sentence explaining what this covers and when it makes sense.]

**Option 3 — [Short title]**
[One sentence explaining what this covers and when it makes sense.]
```

Rules for the "What's Next" block:

- Always present exactly 2–3 options. Never more, never fewer.
- Always mark exactly one option as `✅ Recommended` — the one that best follows the MVP build order and current context.
- Base the recommendation on what was just completed and what the `ROADMAP.md` milestone requires next.
- Keep each option to one sentence — enough to make a decision, not a full explanation.
- Do not ask the user what they want to do — present the options and let them choose or redirect.
- If the user is mid-milestone, the recommended option is always "continue the current milestone".
- If a milestone was just completed, the recommended option is always "start the next milestone".

Example (after completing domain models):

```
## What's Next

**Option 1 — Create PostgreSQL schema and Alembic migrations ✅ Recommended**
This is the next step in the MVP build order and unblocks every service that needs database access.

**Option 2 — Scaffold the full folder structure first**
Useful if you want to see the complete project layout before writing any logic.

**Option 3 — Set up the local Docker Compose dev environment**
Gets Postgres and Redis running locally so you can test migrations immediately after writing them.
```

## Documentation Updates — Keep Docs in Sync with Code

Whenever you make a change that affects the system — new service, schema change, new agent, new dependency, config change, API endpoint, infrastructure addition — you must update the relevant documentation files in the same response. Do not leave docs out of date.

### Which file to update and when

| What changed | Files to update |
| --- | --- |
| New service or component added | `ARCHITECTURE.md` (diagram + services section), `README.md` (architecture diagram if top-level), `SPEC.md` (if it adds or changes a functional requirement) |
| Agent added, removed, or changed | `AGENT_DESIGN.md` (pipeline diagram + agent spec), `ARCHITECTURE.md` (agent responsibilities table) |
| New dependency or library | `TECH_STACK.md` (stack table + dependency list) |
| Database schema change | `ARCHITECTURE.md` (schema section), `SPEC.md` (data model section) |
| New API endpoint | `SPEC.md` (functional requirements), `README.md` if it changes the public surface |
| Security control added or changed | `SECURITY_GUARDRAILS.md`, `SECURITY.md` if it affects the public policy |
| Milestone completed or scope changed | `ROADMAP.md` (check off items, update status) |
| Infrastructure or deployment change | `ARCHITECTURE.md` (deployment model / infrastructure section), `TECH_STACK.md` |
| New environment variable or config key | `TECH_STACK.md`, `CONTRIBUTING.md` (dev setup section) |

### Rules

- Update docs in the same commit as the code change — never as a separate follow-up.
- When updating an `ARCHITECTURE.md` diagram, regenerate the full Mermaid block. Do not make partial edits to diagram lines.
- Keep the `ROADMAP.md` milestone checklist current: check off completed items as work is done.
- If a change makes a section of any doc stale or incorrect, fix or remove that section.
- Do not add placeholder text ("TBD", "coming soon") — either write the content or omit the section.
